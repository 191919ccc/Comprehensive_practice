# 模型状态选择修正说明

## 修改目标

修正首页“模型验证状态”错误显示为 LightGBM 的问题。当前数据库中最新训练版本的 `selection.best_model_code=3`，实际代表 LSTM 是本轮选择出的最佳非基线模型，但前端卡片显示成了 LightGBM。

## 问题原因

后端 `fetchModelComparison()` 原来直接按 `created_at DESC, model_name ASC, metric_name ASC LIMIT 30` 返回 `ml_model_metrics`。

一次训练会写入大量指标，且同一轮训练的 `created_at` 基本相同。排序后前 30 条容易集中在 `ensemble` 和 `lightgbm`，导致 `lstm`、`random_forest` 或 `selection.best_model_code` 指标没有完整返回给前端。

前端 `modelQualityMetric()` 又只从已返回的 `model_comparison` 中选择指标，因此在数据被截断时会误认为 LightGBM 是当前展示模型。

## 修改文件

- `java-backend/src/main/java/com/recruitment/backend/service/DashboardService.java`
- `frontend/index.html`

## 具体修改

1. 后端 `fetchModelComparison()` 改为先定位最新训练版本的 `model_version`，并只返回该版本下的完整指标。
2. 后端返回上限从 30 提高到 240，避免一轮训练的模型指标被截断。
3. 后端排序中将 `selection` 和 `ensemble` 放在前面，保证前端能优先拿到模型选择信息和融合权重。
4. 前端 `modelQualityMetric()` 优先读取 `selection.best_model_code`：
   - `1` 表示 `random_forest`
   - `2` 表示 `lightgbm`
   - `3` 表示 `lstm`
5. 前端找到选中模型后，优先展示该模型的 `walk_forward_balanced_direction_accuracy`，其次回退到 `balanced_direction_accuracy`、`direction_macro_f1`、`direction_accuracy`。

## 验证情况

- 已执行首页内联脚本语法检查：`index inline script ok`。
- 已执行后端编译：`mvn -q -DskipTests compile`。

## 运行说明

需要重启 Java 后端后，前端才会读取修正后的 `/api/dashboard` 返回结果。前端页面刷新后，模型验证状态应优先显示本轮 `selection.best_model_code` 对应的模型。
