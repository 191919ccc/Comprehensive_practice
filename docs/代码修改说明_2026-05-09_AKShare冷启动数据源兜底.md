# 代码修改说明：AKShare 冷启动数据源兜底

## 1. 修改目标

本次修改是为了解决 `python.ml.cold_start` 在批量补齐股票历史日线时全部失败的问题。原脚本主要依赖 AKShare 的东方财富接口 `stock_zh_a_hist`，该接口在当前网络环境下会被远端直接断开，导致批量执行后 `inserted=0`，后续模型无法扩充训练样本。

## 2. 问题现象

执行命令：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --days 2500
```

出现两类错误：

- 开启代理时：`ProxyError('Unable to connect to proxy'...)`
- 关闭代理、清空环境变量后：`RemoteDisconnected('Remote end closed connection without response')`

后者说明请求已经不再走代理，而是东方财富服务器在连接建立后直接关闭连接。因此热点和更新 AKShare 仍然无效，根因更接近东方财富接口对当前请求链路不可用，而不是本机代理没有关闭。

## 3. 根因分析

原代码中 `fetch_a_stock()` 固定调用：

```python
ak.stock_zh_a_hist(...)
```

该函数背后的数据源是东方财富 `push2his.eastmoney.com`。当该接口被远端断开时，脚本没有备用数据源，只能逐只股票失败。

本次测试发现：

- `stock_zh_a_hist` 继续失败；
- `stock_zh_a_hist_tx` 可以正常返回腾讯日线数据；
- `stock_zh_a_daily` 可以正常返回新浪日线数据；
- 腾讯和新浪接口均可返回 600519 的 2019-07-05 至 2026-05-09 区间日线。

## 5. 代码修改内容

### 5.1 增加多数据源参数

新增命令行参数：

```text
--sources
--request-timeout
--retries
--sleep
```

默认数据源顺序改为：

```text
tencent,sina,eastmoney
```

这样当前环境下不会先卡在东方财富，而是优先使用已经验证可用的腾讯数据源。如果以后东方财富恢复，也可以手动指定：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --sources eastmoney,tencent,sina --days 2500
```

### 5.2 增加股票代码市场前缀转换

腾讯和新浪接口要求股票代码带市场前缀，例如：

```text
600519 -> sh600519
000858 -> sz000858
```

因此新增 `market_prefixed_symbol()`，根据 `detect_market()` 自动补 `sh`、`sz` 或 `bj`。

### 5.3 兼容不同数据源字段

原代码只识别东方财富中文字段，例如：

```text
日期、开盘、收盘、最高、最低、成交量、成交额、昨收
```

腾讯和新浪返回的是英文字段，例如：

```text
date、open、close、high、low、volume、amount
```

本次新增 `row_value()` 和 `normalize_trade_date()`，并改造 `normalize_ak_row()`，让同一套入库逻辑可以兼容中文字段和英文字段。

### 5.4 补齐 previous_close 计算

腾讯和新浪日线通常没有 `昨收` 字段。为保证 `change_pct` 可以正常计算，新增 `normalize_stock_frame()`，按时间顺序用上一条收盘价作为当前行的 `previous_close`。

第一条记录没有上一日收盘价时，仍用当前收盘价兜底，避免生成无效行情。

### 5.5 增加数据源失败兜底

新增 `fetch_stock_frame()` 和改造后的 `fetch_a_stock()`：

```text
按 sources 顺序抓取
当前源失败则打印失败原因
继续尝试下一个源
某个源拿到有效记录后立即返回
所有源失败才算该股票失败
```

这样不会因为东方财富一个接口不可用，导致整批股票全部失败。

### 5.6 指数导入也支持兜底

原 `fetch_index()` 只使用东方财富指数接口。现在指数导入也支持腾讯接口兜底，`--include-index --index-only` 已验证可正常导入：

```text
INDEX_SH000001
INDEX_SZ399006
INDEX_SH000300
```

## 6. 验证结果

### 6.1 单只股票验证

执行：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --symbols 600519 --days 60 --request-timeout 5 --sleep 0
```

结果：

```text
[cold-start] fetching 600519
[cold-start] 600519 rows=40 inserted=0
[cold-start] done inserted=0
```

说明默认腾讯源可以直接返回数据。`inserted=0` 是因为该股票对应日期已存在，不是抓取失败。

### 6.2 指数验证

执行：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --include-index --index-only --days 60 --request-timeout 5 --sleep 0
```

结果：

```text
INDEX_SH000001 inserted=40
INDEX_SZ399006 inserted=40
INDEX_SH000300 inserted=40
```

共新增 120 条指数日线。

### 6.3 批量补数结果

批量补数后，`akshare_cold_start` 当前数据量为：

```text
股票数：51
历史日线：83696
时间范围：2019-07-05 至 2026-05-08
```

部分剩余股票仍然抓取很慢，因此没有继续阻塞到全部 92 只补完。当前数据量已经明显高于原先 16 只股票规模，足够重新训练一版模型。

## 8. 当前困境

本次解决了“东方财富不可用导致冷启动全部失败”的问题，但仍有几个现实限制：

1. 批量抓 2500 天历史数据速度较慢，尤其是 90 多只股票一次性抓取时容易超过执行窗口。
2. 腾讯源可用但成交额字段不完整，部分特征会弱于新浪源。
3. 新浪源字段更完整，但 AKShare 文档提示大量抓取容易封 IP，因此不适合无间隔高频批量拉取。
4. 还有 41 只股票没有完成 2500 天全量补齐。
5. 指数数据目前只补了近 60 天，长期指数特征还需要继续分批导入。

## 9. 后续建议

后续不要再直接一次性执行全部股票的 2500 天冷启动。建议分批执行，例如每次 5 到 10 只：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --symbols 601919,601988,601989,000001,000002 --days 2500 --sources sina,tencent --request-timeout 10 --sleep 0
```

如果某批很慢，可以先跳过，等已有数据训练和展示稳定后再补。

当前最优先的下一步不是继续追求一次补完所有股票，而是：

1. 用 51 只股票的新模型观察前端预测是否正常；
2. 再分批补剩余股票；
3. 补更长周期指数数据；
4. 继续比较新旧模型指标，避免盲目覆盖效果更好的版本。
