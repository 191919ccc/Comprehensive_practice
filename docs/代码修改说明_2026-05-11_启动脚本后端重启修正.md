# 启动脚本后端重启修正说明

## 修改目标

修复执行 `tools/start_demo.ps1 -UseRealCrawler` 时，Java 后端打包阶段出现 Spring Boot `repackage` 无法重命名 jar 的问题。

## 问题现象

启动日志中出现：

```text
Unable to rename 'java-backend\target\stock-risk-backend-0.0.1-SNAPSHOT.jar'
to 'java-backend\target\stock-risk-backend-0.0.1-SNAPSHOT.jar.original'
```

## 原因分析

`start_demo.ps1` 每次都会进入 `java-backend` 执行：

```powershell
mvn.cmd -q -DskipTests package
```

如果 8080 端口上已经有旧的 Java 后端进程在运行，该进程可能正在占用 `target\stock-risk-backend-0.0.1-SNAPSHOT.jar`。此时 Spring Boot Maven Plugin 的 `repackage` 步骤需要把原 jar 改名为 `.jar.original`，但文件被占用，所以打包失败。

Kafka、Spark 和 producer 能继续启动，但后端可能仍然是旧进程，前端读到的接口也可能不是最新代码。

## 修改文件

- `tools/start_demo.ps1`

## 具体修改

1. 新增 `Get-ListeningPid()`，用于根据端口查找正在监听的进程 PID。
2. 新增 `Stop-ListeningProcess()`，用于停止占用指定端口的旧进程。
3. 在执行 Maven 打包前，先停止占用 `$BackendPort` 的旧后端进程：

```powershell
Stop-ListeningProcess $BackendPort "backend"
mvn.cmd -q -DskipTests package
```

这样可以释放旧 jar 文件，避免 Spring Boot repackage 阶段重命名失败。

## 验证情况

- 已执行 PowerShell 语法解析检查：

```powershell
[scriptblock]::Create((Get-Content -Raw .\tools\start_demo.ps1))
```

结果：`start_demo.ps1 syntax ok`

## 运行方式

重新执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\start_demo.ps1 -UseRealCrawler
```

如果之前已经启动过 demo，脚本现在会在重新打包前自动停止 8080 上的旧后端进程。
