# DOWN 告警风险分数过滤修改说明

## 修改目的

上一轮 `risk-tight-check` 显示：风险识别标签已经更合理，LightGBM 的 walk-forward 风险识别指标也可用，但 DOWN 告警落地效果仍差：

- DOWN 告警命中率偏低；
- 部分 DOWN 告警后的平均收益为正；
- 说明只靠模型分类为 `DOWN` 还不够，必须叠加真实弱势证据。

本次新增 DOWN 告警风险分数，只有模型看空且近期已有弱势证据时，才允许 `alert_signal=DOWN`。

## 修改文件

- `python/common/config.py`
- `python/.env.example`
- `python/.env`（本地忽略文件）
- `python/ml/stock_ml.py`

## 关键改动

1. 新增配置项

```env
ML_DOWN_ALERT_RISK_SCORE_THRESHOLD=0.55
```

2. 新增 `downside_risk_score()`

风险分数综合以下证据：

- `prob_down`
- `prob_down - prob_flat`
- 近期真实跌幅：`return_1`、`return_3`、`momentum_5`
- 成交量放大下跌：`volume_ratio_20_raw`、`volume_down_breakout`
- 弱于行业或指数：`relative_market_index_return_3`、`relative_market_index_return_5`、`sector_relative_return_5`、`current_vs_market_index_return`

3. 线上预测过滤

`apply_alert_confidence_filter()` 中，风险模式下 DOWN 告警必须同时满足：

- 模型信号为 `DOWN`；
- 置信度通过阈值；
- 预测收益低于 `-ML_RISK_ALERT_MIN_RETURN`；
- `down_risk_score >= ML_DOWN_ALERT_RISK_SCORE_THRESHOLD`；
- 至少有一类真实弱势证据。

否则 `alert_signal` 会被过滤为 `WATCH`，但不改变原始 `predicted_signal`。

4. 训练回测过滤

`alert_backtest_metrics()` 也接入同一套风险分数，避免训练日志里的 DOWN 告警命中率和线上过滤口径不一致。

新增指标：

- `alert_down_risk_score_threshold`
- `alert_backtest_weak_down_evidence_ratio`
- `alert_backtest_down_avg_risk_score`
- `down_alert_risk_score_threshold`

5. 集成预测保留证据字段

`predict_latest_ensemble()` 合并模型结果时保留风险证据字段，确保 ensemble 的 `alert_signal` 也能使用风险分数过滤。

## 判断标准

比上一版更理想的结果应该是：

- `alert_backtest_down_count` 会下降，这是正常的；
- `alert_backtest_down_hit_rate` 应明显上升；
- `alert_backtest_down_avg_inverse_return` 应转为正，或至少不再明显为负；
- `alert_backtest_down_avg_risk_score` 应高于 `0.55`；
- `alert_backtest_weak_down_evidence_ratio` 说明有多少 DOWN 被证据不足过滤。

## 回退方式

如果过滤过严导致 DOWN 告警太少，可以先降低分数阈值：

```bat
set ML_DOWN_ALERT_RISK_SCORE_THRESHOLD=0.45
```

如果要撤销本次代码提交：

```bat
git revert <本次提交ID>
git push origin main
```

## 剩余限制

- 风险分数是规则化过滤，不是新模型；它用于提升 DOWN 告警质量，不直接提升分类指标。
- 如果过滤后仍然命中率低，下一步应把 DOWN 告警改成独立二分类模型，而不是继续依赖方向分类器。
