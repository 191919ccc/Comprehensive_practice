# 代码修改说明：实时流心跳与 Kafka 本机广播修复

## 修改目的

解决实时流演示中的三个稳定性问题：

1. Spark 仍在消费 Kafka 并写入 `metric_snapshots` 心跳时，后端只看 `price_ticks`，容易把实时流误判为延迟或停止。
2. Kafka 没有显式配置 `advertised.listeners`，Spark 可能从 Kafka 元数据里拿到虚拟网卡 IP，换网络后容易出现 Kafka 连接超时。
3. `stop_demo.ps1` 只停止启动脚本记录的包装进程和监听端口进程，旧的 Spark Java 子进程可能残留，干扰下一次实时流启动。

## 修改文件

- `java-backend/src/main/java/com/recruitment/backend/service/DashboardService.java`
- `tools/start_demo.ps1`
- `tools/stop_demo.ps1`
- `java-backend/src/test/java/com/recruitment/backend/service/DashboardServiceTest.java`

## 具体改动

### 1. 后端实时状态同时参考聚合心跳

`/api/dashboard` 的 `stream_status` 不再只使用 `price_ticks.created_at` 判断实时状态，而是同时读取：

- `price_ticks`：真实写入的行情明细。
- `metric_snapshots`：Spark 每个微批写入的聚合心跳。

当行情快照没有变化、`price_ticks` 不新增，但 Spark 仍在写 `metric_snapshots` 时，前端仍能显示实时流处于工作状态。

`/api/health` 的 `stream` 状态也使用同一口径，避免健康检查和首页显示不一致。

### 2. Kafka 运行时配置强制本机广播

`tools/start_demo.ps1` 生成 `E:\stockrt\server-demo.properties` 时，强制写入：

```properties
listeners=PLAINTEXT://127.0.0.1:9092
advertised.listeners=PLAINTEXT://127.0.0.1:9092
```

这样 Spark、Producer 和 Kafka 客户端会稳定连接 `127.0.0.1:9092`，不会被 Kafka 广播到虚拟网卡地址。

### 3. 停止脚本清理演示子进程

`tools/stop_demo.ps1` 增加按命令行识别演示子进程的逻辑，覆盖：

- `stock_streaming_job.py`
- `python.producer.stock_producer`
- `python.producer.stock_replay_producer`
- `stock-risk-backend-0.0.1-SNAPSHOT.jar`
- 前端 `http.server`
- 命令行包含 `E:\stockrt` 的演示运行时进程

这样重新启动项目时不会留下旧 Spark 任务继续消费 Kafka 或连接旧地址。

### 4. 修复后端单测变参匹配

修复 `DashboardServiceTest` 里 `JdbcTemplate.queryForList(String, Object...)` 的 Mockito 变参匹配，保证 `mvn test` 能稳定执行。

## 验证命令

```powershell
$errors = $null; [System.Management.Automation.PSParser]::Tokenize((Get-Content tools\start_demo.ps1 -Raw), [ref]$errors) | Out-Null; if ($errors) { $errors | Format-List; exit 1 } else { 'PowerShell syntax OK' }
mvn -q test
```

重启验证：

```powershell
powershell -ExecutionPolicy Bypass -File tools\stop_demo.ps1
powershell -ExecutionPolicy Bypass -File tools\start_demo.ps1 -UseRealCrawler
```

## 验证结果

- PowerShell 脚本解析通过。
- Java 后端测试通过。
- 重启后 `E:\stockrt\server-demo.properties` 已生成：

```properties
listeners=PLAINTEXT://127.0.0.1:9092
advertised.listeners=PLAINTEXT://127.0.0.1:9092
zookeeper.connect=127.0.0.1:2181
```

- `/api/dashboard` 返回的 `stream_status` 已包含 `latest_price_created_at` 和 `latest_metric_created_at`，并能在价格明细未更新但聚合心跳更新时显示 `FLOWING`。
- `/api/health` 返回的 `stream.status` 已显示 `OK`。

## 剩余限制

如果真实行情源长时间返回完全相同的报价，`price_ticks` 仍可能不新增；这时系统状态会依赖 `metric_snapshots` 心跳判断“实时流仍在工作”。这能解决展示误判，但不能代表外部行情源一定有价格变化。
