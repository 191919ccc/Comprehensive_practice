# 股票实时流分析平台项目说明文档

## 1. 项目概述

本项目是一个面向课程答辩和毕设展示的股票实时流分析平台。系统通过 Python 采集多个股票网站的行情数据，将标准化后的股票行情事件写入 Kafka；Spark Structured Streaming 实时消费 Kafka 数据，完成数据清洗、聚合统计、异常预警识别和 HDFS 归档；MySQL 保存明细、聚合结果、告警记录和机器学习预测结果；Java Spring Boot 后端统一提供 API；前端大屏展示市场指数、价格趋势、AI 预测、风险告警、行业热力图和系统运行状态。

系统主链路如下：

```text
多网站股票行情采集
        ↓
Python Kafka Producer
        ↓
Kafka Topic: stock_realtime_topic
        ↓
Spark Structured Streaming
        ↓
MySQL + HDFS
        ↓
Java Spring Boot API
        ↓
前端可视化大屏
```

项目重点体现三部分能力：

- 实时数据处理：股票行情持续进入 Kafka，并由 Spark 微批处理写入 MySQL 和 HDFS。
- 实时风险识别：根据价格波动和成交量异常阈值生成告警事件。
- AI 辅助分析：基于历史行情训练 Random Forest、LightGBM 和 PyTorch LSTM，生成短期价格预测和方向信号。

## 3. 数据清洗与标准化

### 3.1 采集端清洗

不同网站返回的字段格式不一致，例如 Yahoo、东方财富、新浪和腾讯的字段名称、价格单位、市场代码格式均不同。因此项目在采集端做第一层标准化。

核心函数为：

```text
python/producer/stock_sources.py -> normalize_quote()
```

主要清洗逻辑包括：

- 统一字段名称：将不同网站返回字段统一转换为标准事件字段。
- 统一数值类型：价格转换为 `float`，成交量转换为 `int`。
- 价格保留两位小数：如 `last_price`、`open_price`、`high_price`、`low_price`。
- 计算涨跌幅：`change_pct = (last_price - previous_close) / previous_close * 100`。
- 计算成交额：`turnover = last_price * volume`。
- 自动补充市场和行业：根据股票代码推断 `market` 和 `sector`。
- 数据源兜底：某个网站请求失败时，不中断整轮采集，而是切换到其他数据源。

### 3.2 Spark 端清洗

Spark 流处理入口为：

```text
python/spark/stock_streaming_job.py
```

Spark 从 Kafka 中读取 JSON 字符串，并按照固定 schema 解析为结构化字段。核心流程如下：

```text
Kafka value 原始 JSON
        ↓
from_json 解析
        ↓
字段类型转换
        ↓
event_time 转 timestamp
        ↓
foreachBatch 微批处理
```

Spark 端主要清洗和处理内容包括：

- 按固定 schema 解析 Kafka JSON，避免字段随意变化导致后续统计失败。
- 将 `event_time` 从字符串转换为 `timestamp`。
- 每个微批同时写入 MySQL 和 HDFS。
- 按分钟窗口生成实时统计指标。
- 按股票、行业、分类维度生成聚合结果。
- 根据历史基线和固定阈值识别异常行情。

## 5. 阈值设计

阈值设计是项目中实时风控分析的核心。为了避免只使用固定阈值导致误报，项目采用“固定最低阈值 + 历史动态基线”的组合方式。

### 5.1 历史基线

Spark 每个微批会从 MySQL 的 `price_ticks` 中读取最近一段时间的历史行情，并按股票计算：

```text
avg_abs_change_pct = 历史平均绝对涨跌幅
avg_volume = 历史平均成交量
```

这两个指标作为每只股票自己的历史基线。

为了避免随着 `price_ticks` 增长导致每个微批全表扫描，当前实现只读取最近 1 天的历史行情作为动态基线，并在 `price_ticks` 上设计了 `symbol + event_time`、`symbol + created_at` 等索引。后续如果数据量继续增大，可以把历史基线改为定时任务预计算并缓存到独立表中。

### 5.2 中危告警阈值

价格波动中危阈值：

```text
price_alert_threshold = max(2.0, avg_abs_change_pct * 2.5)
```

含义：

- 如果股票历史波动很小，至少需要达到 2% 才触发价格异常。
- 如果股票本身波动较大，则使用历史平均绝对涨跌幅的 2.5 倍作为阈值。

成交量中危阈值：

```text
volume_alert_threshold = max(market_volume_min_threshold, avg_volume * 2.0)
```

含义：

- 成交量固定下限按市场区分，避免用 A 股标准套用港股和美股。
- 如果股票本身成交量很大，则使用历史平均成交量的 2 倍作为阈值。

市场固定下限：

| 市场 | 中危成交量下限 |
|---|---:|
| SH / SZ / BJ | 5,000,000 |
| HK | 1,000,000 |
| NASDAQ / NYSE / AMEX | 500,000 |
| 其他 | 1,000,000 |

### 5.3 高危告警阈值

价格波动高危阈值：

```text
price_high_threshold = max(4.0, avg_abs_change_pct * 4.0)
```

成交量高危阈值：

```text
volume_high_threshold = max(market_volume_min_threshold * 2.0, avg_volume * 4.0)
```

触发规则：

```text
如果 abs(change_pct) >= price_alert_threshold
或 volume >= volume_alert_threshold
则生成一条告警。

如果 abs(change_pct) >= price_high_threshold
或 volume >= volume_high_threshold
则告警等级为 HIGH。
否则告警等级为 MEDIUM。
```

### 5.4 阈值设计理由

采用动态阈值的原因：

- 不同股票的波动特征不同，不能用同一个绝对标准判断所有股票。
- 大盘股、小盘股、美股、港股和 A 股的成交量级差异明显，需要结合历史成交量。
- 固定最低阈值可以避免历史样本过少时阈值过低导致误报。
- 历史倍数阈值可以让系统对“相对于自身历史显著异常”的股票更敏感。

### 5.5 系统状态阈值

后端还设计了实时链路状态阈值，用于判断系统是否在流动：

| 状态 | 判断条件 | 含义 |
|---|---|---|
| `FLOWING` | 最近 30 秒内有新数据入库 | 实时流正常 |
| `DELAYED` | 最近 5 分钟内有数据，但 30 秒内没有新数据 | 数据流延迟 |
| `STOPPED` | 超过 5 分钟没有新数据 | 流处理或采集可能停止 |

数据源状态阈值：

| 状态 | 判断条件 |
|---|---|
| `OK` | 最近 10 分钟内有该 source 的新数据 |
| `DELAYED` | 最近 60 分钟内有数据，但 10 分钟内没有 |
| `OFFLINE` | 超过 60 分钟没有数据 |

这些阈值用于前端系统状态页展示 Kafka、Spark、采集源和 MySQL 是否正常。

## 6. 模型训练

### 6.1 训练入口

模型训练入口为：

```text
python/ml/train_predict.py
```

运行命令：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.train_predict
```

如果在 Linux 虚拟机或服务器环境运行，使用当前 Python 环境即可：

```bash
python -m python.ml.train_predict
```

训练流程：

```text
从 MySQL price_ticks 读取历史行情
        ↓
构造监督学习样本
        ↓
训练 Random Forest
        ↓
训练 LightGBM
        ↓
训练 PyTorch LSTM
        ↓
生成最新股票预测
        ↓
写入 ml_predictions 和 ml_model_metrics
```

### 6.2 训练数据来源

模型训练数据来自 MySQL 表：

```text
price_ticks
```

读取字段包括：

```text
symbol, company_name, category, sector,
last_price, previous_close, change_pct,
volume, turnover, event_time
```

模型训练不是直接从网页训练，而是从已经经过 Kafka 和 Spark 清洗落库后的行情数据训练。这样可以保证训练数据和实时展示数据来源一致。

### 6.3 标签构造

训练样本按股票代码和时间排序。每一条行情使用同一股票的下一条行情作为预测目标：

```text
next_price = 同一股票下一条行情的 last_price
next_direction = 如果 next_price > last_price，则为 1，否则为 0
next_return = (next_price - last_price) / last_price
```

为避免数据泄露，模型不直接把当前绝对价格作为表格模型输入，也不直接学习“当前价到下一价”的简单线性关系。当前训练目标分为两类：

- 回归任务：预测下一时刻收益率 `next_return`，再用最新价格换算成预测价格。
- 分类任务：预测下一时刻方向，即上涨、下跌或横盘观望。

模型评估采用时间切分，而不是随机切分。即按时间顺序使用前 80% 数据训练，后 20% 数据验证，避免用未来数据训练、过去数据验证导致指标虚高。

### 6.4 特征工程

表格模型使用的特征：

```text
symbol, category, sector,
return_1, volume_change, turnover_change,
ma_5_gap, ma_10_gap, volatility_5,
rsi_14, macd, macd_signal, macd_hist
```

其中：

- `symbol`、`category`、`sector` 为类别特征，使用 One-Hot 编码。
- 数值特征只保留相对变化量和技术指标，不直接使用 `last_price`、`previous_close`、`volume`、`turnover` 等绝对值字段，减少数据泄露和价格尺度记忆问题。

LSTM 使用的时序特征：

```text
return_1
volume_change
turnover_change
ma_5_gap
ma_10_gap
volatility_5
rsi_14
macd
macd_signal
macd_hist
```

这些特征的含义：

| 特征 | 含义 |
|---|---|
| `return_1` | 单步收益率 |
| `volume_change` | 成交量变化率 |
| `turnover_change` | 成交额变化率 |
| `ma_5_gap` | 当前价相对 5 条均线偏离 |
| `ma_10_gap` | 当前价相对 10 条均线偏离 |
| `volatility_5` | 最近 5 条收益率波动率 |
| `rsi_14` | RSI 技术指标 |
| `macd` | MACD 值 |
| `macd_signal` | MACD 信号线 |
| `macd_hist` | MACD 柱状差值 |

### 6.5 当前使用的模型

项目当前训练三类模型：

| 模型 | 用途 | 特点 |
|---|---|---|
| Random Forest | 基线模型 | 稳定、依赖少、便于和其他模型对比 |
| LightGBM | 表格特征预测 | 适合结构化行情数据，训练速度快，效果通常优于随机森林 |
| PyTorch LSTM | 时序预测 | 使用连续行情序列，学习短期趋势变化 |

LSTM 关键配置：

```text
LSTM_WINDOW_SIZE = 20
hidden_size = 64
num_layers = 2
dropout = 0.15
```

`.env` 中的训练配置：

```env
ML_MIN_TRAIN_SIZE=12
ML_MAX_TRAIN_ROWS=24000
ML_REQUIRE_GPU=true
ML_LSTM_EPOCHS=10
ML_LSTM_FINE_TUNE_EPOCHS=2
ML_LSTM_MAX_SEQUENCES=6000
```

含义：

- `ML_MAX_TRAIN_ROWS=24000`：最多使用 24000 条训练样本，避免本机训练过慢。
- `ML_REQUIRE_GPU=true`：要求 PyTorch 使用 GPU 训练。
- `ML_LSTM_EPOCHS=10`：LSTM 训练 10 轮。
- `ML_LSTM_FINE_TUNE_EPOCHS=2`：在全量样本上再微调 2 轮。
- `ML_LSTM_MAX_SEQUENCES=6000`：最多使用 6000 个 LSTM 序列样本。

### 6.6 模型融合

系统会同时训练 Random Forest、LightGBM 和 LSTM。其中 Random Forest 作为 baseline，用于和更强模型做效果对比；真正用于综合预测的是 LightGBM 和 LSTM。

融合逻辑：

- LightGBM 和 LSTM 都存在时，合并两者预测。
- 预测价格按照两个模型的验证集方向准确率加权平均，而不是简单平均。
- 如果方向一致，则输出 `UP` 或 `DOWN`。
- 如果方向不一致，但某个模型验证准确率明显更高，则采用更高准确率模型的方向。
- 如果方向不一致且准确率差距不明显，则输出 `WATCH`，表示观望。
- 如果 LightGBM 或 LSTM 缺失，则退回到当前可用的最佳模型。

预测结果写入：

```text
ml_predictions
```

模型指标写入：

```text
ml_model_metrics
```

常见模型指标包括：

| 指标 | 含义 |
|---|---|
| `price_mae` | 价格预测平均绝对误差 |
| `return_mae` | 收益率预测平均绝对误差 |
| `direction_accuracy` | 涨跌方向预测准确率 |

请求体示例：

```json
{
  "status": "ACKED",
  "note": "已确认该股票短时放量",
  "handled_by": "user"
}
```

说明：当前 Spark 实时告警只实现了 `price_volatility` 和 `volume_spike` 两类规则。MACD、RSI、均线偏离等指标目前用于机器学习特征工程和趋势分析，不作为 Spark 告警规则直接触发条件。后续如需展示“MACD 背离、RSI 超买、均线死叉”等技术告警，需要在 Spark 端增加对应实时计算逻辑。

## 8. 当前方案的说明与限制

当前项目已经具备完整实时流闭环，但仍有一些需要向老师说明的限制：

- 股票网站免费接口可能存在访问频率限制，真实采集速度不能无限提高。
- 股市休市时，真实行情变化较少，前端数据变化会变慢。
- 当前训练数据量仍偏少，LSTM 更适合在连续采集多天或导入历史行情后使用；数据不足时需要主动说明模型结果只作为课程演示级参考。
- 如果训练数据主要来自历史回放或短时间内重复采集，验证集准确率可能异常偏高；答辩时应说明这是演示数据分布较简单导致的结果，真实效果需要更长周期、更复杂市场环境的数据验证。
- 模型训练不是每 5 秒执行一次，而是定时或手动训练；实时性主要体现在新行情入库后的实时计算、告警和展示。
- 当前预测为课程项目级辅助分析，不应作为真实投资建议。
- 前端“AI 解读”和“生成完整分析报告”是基于现有指标生成分析提示词，真正的投资结论仍需要人工判断。
- 当前 Spark 实时统计特征和机器学习训练特征还不是完全统一的一套特征服务。Spark 负责实时聚合、异常阈值和落库，训练脚本从 `price_ticks` 重新计算 RSI、MACD、均线偏离等模型特征。答辩时可说明这是第一版工程取舍，后续会把特征计算抽成统一模块或特征表，避免实时推理和离线训练口径不一致。

## 9. 后续优化方向

后续可以继续增强以下功能：

- 增加真实指数采集：单独采集上证指数、深证成指、创业板指，而不是用市场概览模拟。
- 增加模型定时训练：使用 Windows 任务计划或脚本定时运行 `python.ml.train_predict`。

## 10. 模型融合修正说明

本项目当前将三个模型的职责重新划分为：

- Random Forest：只作为 baseline 基线模型，用于判断特征工程是否有效，不参与最终融合预测。
- LightGBM：负责结构化表格特征预测，适合处理行业、分类、相对收益率、成交量变化率、技术指标等截面特征。
- PyTorch LSTM：负责时序模式预测，使用最近 20 条连续行情序列学习短期动量和趋势变化。

为避免时序数据评估虚高，训练集和验证集不再随机切分，而是按每只股票内部的时间顺序分别做 8:2 切分。这样同一只股票的未来行情不会进入训练集后再去验证过去行情。

方向标签不再把价格不变的样本强行标记为下跌，也不再过滤横盘样本。系统采用三分类方向标签：

| 标签 | 含义 | 前端展示 |
|---|---|---|
| `UP` | 下一时刻收益率大于最小波动阈值 | 看多 |
| `DOWN` | 下一时刻收益率小于负的最小波动阈值 | 看空 |
| `FLAT` | 下一时刻价格基本不变 | 观望 / WATCH |

这样模型既能学习价格真实波动时的上涨和下跌，也能学习休市、低流动性或重复推送时的横盘状态，避免选择偏差。

最终融合只使用 LightGBM 和 LSTM。系统先在验证集上获得两个模型的方向准确率，再归一化为融合权重：

```text
lightgbm_weight = lightgbm_direction_accuracy / (lightgbm_direction_accuracy + lstm_direction_accuracy)
lstm_weight = lstm_direction_accuracy / (lightgbm_direction_accuracy + lstm_direction_accuracy)
final_probability[class] = lightgbm_weight * lightgbm_probability[class] + lstm_weight * lstm_probability[class]
```

方向信号选择：

```text
argmax(final_probability[DOWN], final_probability[UP], final_probability[FLAT])
DOWN => 看空
UP   => 看多
FLAT => WATCH / 观望
```

置信度字段 `confidence` 的定义：

```text
raw_certainty = max(final_probability[DOWN], final_probability[UP], final_probability[FLAT])
confidence = 0.5 + normalize(raw_certainty) * 0.5
```

`confidence` 写入 MySQL 时已经映射到 0.5 到 1.0 的展示区间。前端显示为 50% 到 100%，更适合课堂展示和普通用户理解。越接近 100%，说明模型对当前 `UP / DOWN / WATCH` 信号越确定；越接近 50%，说明模型对三个方向区分度较低。

## 11. 冷启动、漂移检测与工程细节

### 11.1 冷启动历史数据预热

实时系统刚启动时，Kafka 中的实时 tick 数量较少，模型即使满足 `ML_MIN_TRAIN_SIZE=12` 也没有足够样本学习有效规律。因此项目增加冷启动导入脚本：

```bash
python -m python.ml.cold_start --days 900
```

Windows 本机运行：

```powershell
D:\anaconda3\envs\MachineLearn\python.exe -m python.ml.cold_start --days 900
```

该脚本通过 AKShare 拉取 A 股历史日线行情，转换为统一的 `price_ticks` 结构后批量写入 MySQL，作为模型训练底仓。它不会替代实时采集，作用是让系统启动初期也有足够历史数据可训练。

说明：

- 该功能依赖 `akshare`，如环境缺失需先执行 `pip install akshare`。
- 冷启动数据的 `source` 字段为 `akshare_cold_start`。
- 插入前会按 `event_id = cold-akshare-{symbol}-{date}` 和 `(source, symbol, event_time)` 双重检查是否已存在，避免旧入口和新入口重复导入。

### 11.2 模型漂移检测

项目新增 `ml_prediction_history` 表保存每次训练产生的预测历史，避免 `ml_predictions` 只保留最新结果导致无法回溯模型效果。

漂移检测入口：

```bash
python -m python.ml.model_drift --window 50 --threshold 0.55
```

逻辑：

- 从 `ml_prediction_history` 读取最近预测记录。
- 找到预测时间之后同一股票的第一条真实行情。
- 比较预测方向和实际方向，计算最近窗口准确率。
- 若准确率低于阈值，向 `alert_events` 写入 `model_drift` 告警。
- 同时向 `ml_model_metrics` 写入 `drift_monitor / recent_direction_accuracy` 指标。

训练脚本 `python.ml.train_predict` 在写入预测结果后会自动执行一次漂移检测。样本不足时只返回未就绪，不会误报。

### 11.3 Kafka 分区策略

项目生产者已经使用股票代码作为 Kafka key：

```python
producer.send(settings.kafka_topic, key=quote["symbol"], value=quote)
```

这样同一股票的事件会尽量进入同一分区，有利于保持单只股票的事件顺序，并方便 Spark 后续按股票维度处理。

### 11.4 MySQL 索引

`price_ticks` 已包含以下关键索引：

```sql
KEY idx_price_ticks_symbol_event_time (symbol, event_time)
KEY idx_price_ticks_symbol_created_at (symbol, created_at)
```

这些索引用于支撑模型训练按股票和时间排序、后端查询单只股票趋势、Spark 回查历史基线。

### 11.5 异常 tick 过滤

项目新增统一的脏数据过滤逻辑，避免真实行情接口偶发异常值污染告警和模型：

- `last_price <= 0` 过滤。
- `volume <= 0` 过滤。
- A 股涨跌幅绝对值超过 11% 过滤。
- 港股涨跌幅绝对值超过 50% 过滤。
- 其他市场涨跌幅绝对值超过 80% 过滤。

过滤位置：

- 真实采集端：`python/producer/stock_producer.py`
- 历史回放端：`python/producer/stock_replay_producer.py`
- Spark 实时清洗端：`python/spark/stock_streaming_job.py`
